<Subtitle>Keep the Python package, architecture rules, agent skills, and published documentation in agreement.</Subtitle>

## Start with the repository contract

Read the root `AGENTS.md` before editing. Specx publishes two connected products from one repository:
the typed Python guardrail package under `src/specx/` and the canonical agent catalog under `skills/`.
Tests and helper scripts live under `tests/` and `scripts/`.

Install the locked development environment, then establish a clean baseline:

```sh
uv sync --locked --all-groups
npm ci
make check
npm run docs:check
```

## Change a skill

Edit only the canonical `skills/<skill-name>/` directory. Keep `SKILL.md` concise and
trigger-oriented; put reusable patterns and longer examples in `references/`. A skill frontmatter
contains only `name` and `description`, and its name matches the directory.

After editing, synchronize the tracked local-discovery mirror and validate it:

```sh
make sync-skills
make validate-skills
make list-skills
```

Do not edit `.agents/skills/` by hand. If a reference exceeds 100 lines, give it a short linked
contents section. UI metadata in `agents/openai.yaml` must keep its prompt and description aligned
with the canonical skill.

## Change an architecture rule

Keep one stable semantic rule ID. Update the implementation and focused tests, then update every
skill reference that teaches the same behavior. If the rule changes generated-project expectations,
update the root agent guidance and compatibility renderer as well.

```sh
uv run specx rule explain use-cases.return-dtos
make test
make check
```

Forward-test substantial generation changes in a disposable initialized project when practical.
Rules are guardrails for the architecture described by the skills; neither side should silently
invent a different contract.

## Change these docs

Documentation pages are MDX files under `docs/`. Add each page to `.storybook/docsNavigation.ts`
and the inline Storybook order in `.storybook/preview.ts`. Every page except Introduction ends with
`NextPrev`, and internal links use Storybook docs paths.

```sh
npm run docs:check
npm run types
npm run build-storybook -- --output-dir storybook-static
npm run docs:check-built
```

Inspect the built site in light and dark themes and at mobile width. Tables and code samples must
scroll locally instead of widening the page. Run `make check` before opening or updating a pull
request.

## Pull request checklist

- Package behavior, skill instructions, and architecture references agree.
- `make check` and the documentation checks pass.
- Generated-project changes were exercised outside this repository when practical.
- No placeholder directories, copied foundation bases, caches, or local environment files are added.
- User-facing prose describes observable behavior and real commands without promotional filler.
