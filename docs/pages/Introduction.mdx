<Title>specx</Title>
<Subtitle>Agent skills and executable architecture rules for Python services.</Subtitle>

specx gives coding agents a concrete way to build and change Python backends without letting
framework, persistence, and application concerns drift together. It combines four surfaces:

- **Agent skills** describe how to structure projects and implement common backend work.
- **The project initializer** renders a strict framework-neutral starting point.
- **Foundation classes** give project classes explicit architectural categories.
- **Architecture checks** inspect the result and report semantic rule IDs.

The skills guide the change. The checks inspect the result. Neither replaces normal design review,
tests, or framework documentation.

## Where specx fits

Use specx when a Python service needs explicit application boundaries that coding agents can follow
and CI can verify. The catalog covers framework-neutral cores, FastAPI delivery, DIWire composition,
SQLAlchemy and Alembic, runtime settings, external adapters, and test structure.

specx is deliberately opinionated. Use cases take one command or query and return DTOs. Transactional
work enters through a unit-of-work manager. Delivery and infrastructure stay outside the inner core.
Project classes inherit a foundation category that states their role.

If those constraints do not match the service you want to build, adopt individual ideas rather than
disabling most of the rule set.

## Typical ways to use it

- Start a new service with `specx init`, then ask an agent to apply the relevant specx skill.
- Install the Python package in an existing `src/`-layout service and introduce guardrails gradually.
- Use the foundation classes without the skills when humans own the implementation workflow.
- Install the skills without the package for design guidance, then add executable checks when the
  project is ready.
