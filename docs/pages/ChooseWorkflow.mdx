<Title>Choose a workflow</Title>
<Subtitle>Start from the change you need, not from the layer names.</Subtitle>

Use one primary skill for a focused change. Combine skills when the work crosses real boundaries—for
example, a persisted HTTP feature usually needs a use case, an adapter, a migration, a controller,
composition, and tests.

## Common combinations

### A framework-neutral domain action

Use `specx-add-core-use-case`, then `specx-add-core-service` only if the action contains reusable
behavior. Add `specx-tests` for the unit test shape.

### A persisted FastAPI endpoint

Start with `specx-component-architecture` if the scope is new. Then use the use-case, infrastructure,
migration, delivery, composition, settings, and testing skills for their respective parts. The use
case owns the transaction boundary; the controller owns HTTP mapping.

### An external API integration

Use `specx-add-infrastructure-adapter` to define a business-language gateway and its technical
adapter. Use `specx-settings` for credentials and endpoints. Inject the gateway port into core code,
not the SDK client or settings object.

### An existing service with mixed boundaries

Use `specx-component-architecture` to map the current structure before moving files. Add the Python
package, configure a small selector set, fix violations by boundary, then broaden selection. See
[Adopt Specx in an existing service](?path=/docs/guides-adopt-specx-in-an-existing-service--docs).

## When not to add another abstraction

Do not create an interface only to make a concrete collaborator mockable. Use a port for external IO
or for multiple real implementations. Do not turn every helper into a service; a narrow replaceable
ability may be a capability, while a local pure function may need no class at all unless project
policy requires one.
