## `uow.ioc-does-not-register-active-uow`

Reject active unit-of-work registrations in the IOC container.

**Status:** Enabled by default.

### Why this exists

An active unit of work is transaction-scoped state and must not be registered as a reusable container dependency.

### Failing example

```python
from diwire import Lifetime

container.add(SQLAlchemyOrderUnitOfWork, provides=OrderUnitOfWork)
```

### Passing example

```python
container.add(
    SQLAlchemyOrderUnitOfWorkManager,
    provides=OrderUnitOfWorkManager,
)
```

### Detection notes

Checks `ioc/container.py` for `Lifetime` imports and container registrations whose explicit `provides=` type or inferred positional target ends in `UnitOfWork` rather than `UnitOfWorkManager`. This covers `add`, `add_factory`, and `add_instance` forms.
