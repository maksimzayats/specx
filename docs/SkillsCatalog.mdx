## Project and architecture

| Skill | Use it when | Main output |
| --- | --- | --- |
| `specx-project-structure` | Starting or broadly reshaping a FastAPI service | Real vertical-slice structure, root guidance, delivery and probe ownership |
| `specx-component-architecture` | A feature crosses layers or its class category is unclear | Placement and dependency decisions before broad edits |
| `specx-foundation` | Choosing a packaged base or adding a justified local category | Explicit foundation ancestry without copied or empty base packages |
| `specx-project-tooling` | Adding uv, Ruff, mypy, pytest, Make targets, or validation policy | Strict executable tooling aligned with `AGENTS.md` |

For a new framework-neutral project, run `specx init`. The project-structure skill specifically
targets a FastAPI service and enables the FastAPI rule family when that delivery exists.

## Core behavior

| Skill | Use it when | Main output |
| --- | --- | --- |
| `specx-add-core-use-case` | Adding one externally meaningful command or query | Same-file input, DTO result, explicit transaction ownership, unit tests |
| `specx-add-core-service` | Extracting reusable application behavior | Pure, read, or effect service with focused tests |

The component-architecture skill is a useful prerequisite when the work introduces a new scope or
when “service,” “capability,” “repository,” and “gateway” are competing interpretations.

## Technical edges

| Skill | Use it when | Main output |
| --- | --- | --- |
| `specx-add-infrastructure-adapter` | Talking to databases, caches, files, queues, HTTP APIs, or SDKs | Core port, scope adapter, resource ownership, binding, integration tests |
| `specx-add-delivery-controller` | Adding FastAPI routes, schemas, delivery helpers, or lifecycle | Top-level controller/schema mapping and HTTP integration tests |
| `specx-diwire-composition` | Wiring ports, managers, settings, clients, factories, or test overrides | Explicit bindings at `ioc` and clean constructor injection |

| `specx-sqlalchemy-migrations` | Adding SQLAlchemy models or repairing migration ownership | Async Alembic setup, project metadata, discovery, revisions, drift tests |

## Tests

| Skill | Use it when | Main output |
| --- | --- | --- |
| `specx-tests` | Adding unit, integration, delivery, migration, end-to-end, or custom guardrail tests | Mirrored flat tests, native container fixtures, correctly placed doubles |

## Skill contract

Each skill directory contains:

- `SKILL.md` with trigger-oriented instructions;
- `references/*.md` with detailed patterns and examples;
- `agents/openai.yaml` with agent UI metadata when present.

The canonical catalog is under [`skills/`](https://github.com/maksimzayats/specx/tree/main/skills).
The repository validates frontmatter, naming, reference structure, metadata, and the tracked local
discovery mirror. Skills are guidance: they do not run generators, mutate projects when installed, or
replace review of the resulting code.
